# EEP-0003 — Share Extension (EXT-19)

> **EEP**: 0003
> **标题**: Share Extension — Link-based Resource Sharing
> **类型**: Standards
> **状态**: Draft
> **作者**: Allen
> **创建日期**: 2026-03-01
> **关联 Spec 版本**: v0.9.5
> **目标 Spec 版本**: TBD
> **依赖 EEP**: [EEP-0001](EEP-0001.md)（ezagent URI Scheme）

---

## Abstract

提出 EXT-19 Share Extension，在 EEP-0001 URI Scheme 基础上增加 **带权限控制的资源分享** 能力。用户或 Socialware 创建 Share Link 后，接收方点击链接即可在 ezagent 客户端中打开目标资源，并根据权限执行对应操作（查看、加入、安装、Fork 等）。

Share Link 支持丰富的权限模型（过期时间、使用次数、可见范围），Socialware 开发者通过 `@shareable` 编程式声明控制哪些资源可分享以及分享时如何转换/脱敏。

## Motivation

### 现状问题

1. **无标准分享机制**。用户想把一段讨论、一个 Room 结构、一个 Agent 配置分享给另一个实例上的同事，没有协议层支持。只能口头描述或手动复制。
2. **Socialware 资源不可传递**。TaskArena 的任务模板、EventWeaver 的事件结构、AgentForge 的 Agent 配置——这些有价值的资产无法作为可传递的单元在实例间流动。
3. **"组织即代码" 的 Fork/Compose 缺少传输层**。socialware-spec §4 定义了 Fork 等组合操作的语义，但没有定义"Socialware 声明如何从 A 实例到达 B 实例"。
4. **没有权限控制的分享是危险的**。聊天记录、组织配置可能包含敏感信息，分享时需要控制谁能看、看多久、看哪部分。

### 与 Fork/Compose/Merge 的关系

Share 和 Fork 是**独立平行**的两个机制：

| | Fork | Share |
|---|---|---|
| 语义 | 创建活的新实例（CRDT 同步） | 生成可传递的访问入口 |
| 数据流 | 实时同步 | 按需获取（点击时拉取） |
| 类比 | Git `clone`（活连接） | 发送一个 GitHub repo 链接 |
| 是否需要双方同时在线 | 是 | 否（Relay 可代持） |

但两者可以协作：Share Link 的一种 Action 是 "Fork this Socialware"——用户点击 Share Link → 预览结构 → 确认 Fork → 触发 socialware-spec §4.1 的 Fork 操作。

## Design

### D1. Share Link 格式

Share Link 是一个指向 `/s/{token}` 保留路径的 ezagent URI（EEP-0001 §D2）：

```
ezagent://relay.example.com/s/{share_token}
https://relay.example.com/e/s/{share_token}         ← Web fallback
```

**share_token 格式**：URL-safe Base64 编码的 22 字符随机串（128-bit entropy）。

```
示例：ezagent://relay.example.com/s/dGhpcyBpcyBhIHRva2Vu
```

- [MUST] share_token MUST 具有至少 128-bit entropy
- [MUST] share_token MUST 为 URL-safe 字符集（`[A-Za-z0-9_-]`）
- [MUST] share_token 一经生成不可修改，撤销通过 revoke 操作实现

### D2. Share Record

每个 Share Link 对应一条 Share Record，存储为 Room 内的 Message（content_type: `share:record`）：

```yaml
content_type: "share:record"
body:
  # ── 标识 ──
  share_id:     string              # UUIDv7，Share Record 唯一标识
  token:        string              # share_token（索引用，不含 authority）

  # ── 指向什么 ──
  target:
    uri:        string              # 目标资源的 ezagent URI（EEP-0001 格式）
    type:       enum                # room | message | timeline | socialware | agent | extension | composite
    snapshot:   boolean             # true: 分享创建时刻的快照; false: 实时数据

  # ── 权限控制 ──
  policy:
    access:     enum                # public | authenticated | invited
    invited:    [string]            # access=invited 时，允许的 Entity ID 列表
    expires_at: datetime | null     # 过期时间（null=永不过期）
    max_uses:   integer | null      # 最大使用次数（null=无限）
    uses:       integer             # 当前已使用次数（State Cache 派生）

    # ── 可见范围 ──
    scope:
      channels:      [string] | null   # 仅这些 Channel 可见（null=全部）
      content_types: [string] | null   # 仅这些消息类型可见（null=全部）
      time_range:                      # 仅此时间范围（null=全部）
        after:  datetime | null
        before: datetime | null
      redactions:    [string]          # 需脱敏的字段路径（如 "body.email"）

    # ── 允许的操作 ──
    actions:    [enum]              # view | join | fork | install | clone_config
                                    # 接收方可以执行哪些操作

  # ── 元信息 ──
  created_by:   string              # 创建者 Entity ID
  label:        string              # 人类可读描述（如 "Q1 Review 讨论记录"）
  revoked:      boolean             # 是否已撤销
  revoked_at:   datetime | null
  revoked_by:   string | null
```

**access 三种模式**：

| 模式 | 行为 |
|------|------|
| `public` | 任何人（含无 ezagent 账户的 Web 访客）可访问 |
| `authenticated` | 任何持有 ezagent Identity 的用户可访问 |
| `invited` | 仅 `invited` 列表中的 Entity 可访问 |

**target.type 与 actions 的对应关系**：

| target.type | 可用 actions | 说明 |
|-------------|-------------|------|
| `room` | view, join, clone_config | 查看 Room 信息 / 加入 Room / 复制 Room 结构 |
| `message` | view | 查看单条消息及上下文 |
| `timeline` | view | 查看时间线片段 |
| `socialware` | view, fork, install | 查看 SW 结构 / Fork 为新实例 / 安装到本地 |
| `agent` | view, install | 查看 Agent 配置 / 安装到本地 |
| `extension` | view, install | 查看 Extension 信息 / 安装 |
| `composite` | view, *（取决于子项） | 组合分享，每个子项独立 action |

### D3. Composite Share（复合分享）

一个 Share Link 可以指向多个资源的组合：

```yaml
target:
  type: "composite"
  uri: null                          # composite 无单一 URI
  items:
    - uri: "ezagent://relay.example.com/r/{room_id}"
      type: "room"
      label: "项目讨论室"
    - uri: "ezagent://relay.example.com/r/{room_id}/sw/ta"
      type: "socialware"
      label: "TaskArena 配置"
    - uri: "ezagent://relay.example.com/r/{room_id}/sw/af/agent/review-bot"
      type: "agent"
      label: "Review-Bot 配置"
```

接收方打开 Composite Share 后看到资源清单，可逐项或批量执行 action。

- [MUST] Composite Share 中每个 item MUST 独立遵守 policy 的权限控制
- [MUST] 接收方对每个 item 的 action MUST 独立确认（不得一键全部执行）

### D4. Share 生命周期

```
Create ──→ Active ──→ Expired / Exhausted
              │
              └──→ Revoked

Active:     可正常使用
Expired:    expires_at 已过
Exhausted:  uses >= max_uses
Revoked:    创建者或 admin 主动撤销
```

**状态派生**（State Cache，从 Message 序列计算）：

```python
def share_status(record):
    if record.revoked:
        return "revoked"
    if record.policy.expires_at and now() > record.policy.expires_at:
        return "expired"
    if record.policy.max_uses and record.uses >= record.policy.max_uses:
        return "exhausted"
    return "active"
```

- [MUST] 每次 Share Link 被使用时，MUST 发送 `share:use` Message 记录使用事件
- [MUST] `uses` 计数从 `share:use` Message 序列派生，不直接修改 Share Record

### D5. Hooks

| Hook | Phase | Trigger | Priority | 说明 |
|------|-------|---------|----------|------|
| `share.validate_access` | pre_send | share resolve 请求 | 40 | 验证 token 有效性、权限、配额 |
| `share.apply_scope` | after_read | 读取共享资源数据 | 45 | 根据 policy.scope 过滤不可见内容 |
| `share.record_use` | after_write | share:use insert | 50 | 更新 State Cache 中的 uses 计数 |
| `share.notify_creator` | after_write | share:use insert | 55 | 可选：通知创建者有人使用了链接 |
| `share.transform` | after_read | 读取共享资源数据 | 60 | 调用 Socialware 注册的 `@shareable` transform |

**after_read 阶段的处理链**：

```
原始数据 → share.apply_scope（过滤 channel/content_type/time_range）
         → share.transform（Socialware 自定义脱敏/转换）
         → 返回给接收方
```

### D6. Share Resolver（数据获取流程）

接收方点击 Share Link 后的完整流程：

```
接收方客户端                        Relay                    源 Peer
──────────                         ─────                    ───────
1. 解析 URI
   authority = relay.example.com
   path = /s/{token}
          │
2. 向 Relay 发送 resolve 请求 ──→  3. 查找 Share Record
   (携带自身 Entity ID)                │
                                      ├─ token 无效 → 404
                                      ├─ 权限不足 → 403
                                      ├─ 已过期/耗尽 → 410
                                      │
                                      4. 返回 Share Record
                                         + 源 Peer 定位信息
          │
5. 展示预览（resource 描述、
   可用 actions、权限范围）
          │
6. 用户选择 action ─────────────→  7. Relay 转发或
   (view/join/fork/install)            提供 P2P 连接信息
          │                                                   │
8. ←──────────── P2P 直连获取数据 ──────────────────────────→ │
   或通过 Relay 中转                                          │
          │
9. 执行 action
   - view: 只读展示（经 scope + transform 过滤）
   - join: 调用 room.join
   - fork: 触发 socialware-spec §4.1 Fork
   - install: 复制 manifest + files 到本地
   - clone_config: 复制 Room Config 创建新 Room
```

- [MUST] 步骤 5（预览）MUST 在执行 action 前展示，用户 MUST 确认后才执行
- [MUST] Relay 在步骤 3 验证权限时 MUST 调用 `share.validate_access` Hook
- [SHOULD] 实际数据传输 SHOULD 优先走 P2P 直连

### D7. `@shareable` 声明（Socialware Developer DSL）

Socialware 开发者通过 `@shareable` 装饰器控制其资源的分享行为：

```python
@socialware("task-arena")
class TaskArena:

    @shareable(
        content_types=["ta:task.*", "ta:submission.*"],
        actions=["view", "fork"],
    )
    async def share_defaults(self, ctx):
        """声明 TaskArena 的默认可分享配置"""
        pass

    @shareable.filter(content_types=["ta:task.*"])
    async def filter_tasks(self, ref, ctx, share_policy):
        """决定单条 task 是否可被纳入 share"""
        # 仅分享已完成的任务
        task_state = await ctx.state_cache.get(f"ta:task:{ref.ref_id}")
        return task_state.status in ("approved", "archived")

    @shareable.transform(content_types=["ta:task.*"])
    async def redact_task(self, ref, ctx, share_policy):
        """分享时对 task 内容做脱敏转换"""
        body = dict(ref.body)
        # 移除内部评审意见
        body.pop("internal_notes", None)
        # 匿名化 worker 信息（如 policy 中要求）
        if "worker_identity" in share_policy.scope.redactions:
            body["claimed_by"] = "[anonymous]"
        return body
```

**`@shareable` API**：

```python
@shareable(
    content_types: list[str],          # 哪些 content_type 可分享（支持通配符 *）
    actions: list[str],                # 允许的 actions
    default_scope: dict | None = None, # 默认 scope 覆盖
)

@shareable.filter(content_types: list[str])
async def filter(self, ref, ctx, share_policy) -> bool:
    """返回 True 表示该 Ref 可纳入分享，False 排除"""

@shareable.transform(content_types: list[str])
async def transform(self, ref, ctx, share_policy) -> dict:
    """返回转换后的 body，用于分享视图"""
```

**执行时机**：

- `@shareable()` 声明在 Socialware 加载时注册到 Share Extension
- `@shareable.filter` 在 Share 创建时调用（决定哪些 Ref 进入 Share 范围）
- `@shareable.transform` 在 Share 被访问时调用（`share.transform` Hook 内）

**未注册 `@shareable` 的 Socialware**：

- [MUST] 如果一个 Socialware 没有声明 `@shareable`，其所有 content_type 默认 **不可被 Share** 纳入
- [MUST] Bus 内置的 content_type（`immutable`）默认可分享，无需 `@shareable` 声明
- [MUST] Extension 注册的 content_type（`mutable`、`collab` 等）默认可分享

### D8. Room Config

```yaml
ext.share:
  enabled: true                          # 是否允许在此 Room 创建 Share
  default_policy:                        # Room 级别的默认分享策略
    access: "authenticated"
    expires_after: "7d"                  # 默认 7 天后过期
    max_uses: null
    actions: ["view"]
  restrictions:
    allow_public: false                  # 是否允许创建 public Share
    allow_composite: true                # 是否允许 Composite Share
    min_power_level: 50                  # 创建 Share 所需的最低 power_level
```

- [MUST] `ext.share` 的 writer_rule: `signer.power_level >= admin`
- [MUST] Room Config 的 restrictions 优先于个人创建 Share 时的请求

### D9. CLI 命令

```bash
# 创建 Share
ezagent share create RESOURCE_URI \
  --access authenticated \
  --expires 7d \
  --max-uses 10 \
  --actions view,fork \
  --channels general,announcements \    # scope: 仅这些 channel
  --label "Q1 Review 讨论记录"

# 输出: ezagent://relay.example.com/s/dGhpcyBpcyBhIHRva2Vu

# Composite Share
ezagent share create \
  --items "ezagent://.../r/{room_id}" "ezagent://.../r/{room_id}/sw/ta" \
  --access invited \
  --invite @bob:relay.example.com @carol:relay.example.com

# 管理
ezagent share list                       # 列出我创建的 Share
ezagent share info {token}               # 查看 Share 详情
ezagent share revoke {token}             # 撤销
ezagent share stats {token}              # 使用统计

# 使用（打开别人的 Share）
ezagent open ezagent://relay.example.com/s/{token}
```

## Alternatives Considered

### A1. 文件打包导出（ShareBundle）

将资源序列化为 `.ezshare` 文件，通过文件传递。

**优点**：完全离线可用；可通过邮件/网盘传递。

**放弃原因**：违反 ezagent 数据活在 CRDT 网络中的核心理念；文件快照立即过时；权限控制困难（文件一旦发出无法撤回）；用户体验差（下载→导入 vs 点击链接→直接看到）。

**未来考虑**：如果有明确需求（如完全离线场景），可作为 Share Extension 的子功能追加，将 link resolve 的结果序列化为文件。不需要独立 EEP。

### A2. 基于 Room 邀请机制扩展

复用 Room 的 `room.invite` 操作，通过邀请 Entity 加入包含目标内容的 Room 来实现分享。

**优点**：零新概念，复用现有机制。

**放弃原因**：只能分享 Room 级别的资源（无法分享单条消息、Timeline 片段、Agent 配置等）；无法实现"只读查看"语义（join Room 后获得完整权限）；无过期/次数控制。

### A3. 声明式 @shareable（manifest.toml 中静态配置）

在 Socialware manifest.toml 中用 TOML 声明可分享的 content_type 列表。

**优点**：简单、无运行时开销。

**放弃原因**：无法实现动态 filter（如"仅分享已完成的任务"）；无法实现 transform/脱敏；不同 Share 实例可能需要不同的 filter 逻辑。编程式声明（`@shareable`）可以覆盖声明式的所有场景，反之不行。

**验收标准**：

- [ ] 创建 Share Link 后，另一个实例的用户可点击打开并预览资源
- [ ] expired / exhausted / revoked 的 Share Link 返回对应错误
- [ ] scope 过滤正确（仅可见指定 channel/content_type/time_range 的内容）
- [ ] `@shareable.filter` 正确排除不可分享的 Ref
- [ ] `@shareable.transform` 正确脱敏返回内容
- [ ] Composite Share 中各 item 独立验权
- [ ] Web Fallback URL 在浏览器中可展示只读预览

## Open Questions

### Q1. Share Record 存储位置

Share Record 作为 Message 存储在哪个 Room？

- **选项 A**：存储在目标资源所在的 Room——自然关联，但 Room 成员都能看到 Share Record
- **选项 B**：存储在创建者的私有 "Share Registry" Room——隔离性好，但增加了一个新 Room 概念
- **选项 C**：存储在 Relay 侧（不走 CRDT）——简单，但违反 "CRDT 是真相源" 原则

倾向 A，配合 power_level 控制 Share Record 的可见性。

### Q2. Relay 的 Share Token 索引

Relay 需要维护 token → Share Record 的索引以支持快速 resolve。这个索引如何与 CRDT 数据同步？

- Relay 作为 Zenoh router 已经收到所有 `share:record` Message 的 update，可以被动构建索引
- 但 Relay 是否应该成为 Share Token 的"权威来源"？还是只是缓存？

### Q3. 跨 Relay 域的 Share

如果 Share 创建者在 `relay-a.com`，目标资源在 `relay-b.com` 管辖的 Room 中，Link 的 authority 应该是哪个？

- **选项 A**：资源所在的 relay_domain（`relay-b.com`）——语义清晰
- **选项 B**：创建者的 relay_domain（`relay-a.com`）——创建者控制生命周期

### Q4. Share Link 的可发现性

如何防止 Share Token 被暴力枚举？128-bit entropy 理论上足够，但是否需要额外的 rate limiting？

### Q5. `@shareable.transform` 的性能

每次 Share 被访问时都调用 Python transform 函数，对于大量 Ref 的 Timeline 分享可能有性能问题。是否需要缓存机制？

### Q6. 分享聊天内容时的身份问题

分享一段 Timeline 时，消息作者的 Entity ID 会暴露给接收方。是否需要提供"匿名化模式"，将所有作者替换为 `Participant 1, 2, ...`？
