## Abstract

为 ezagent 协议定义统一的 URI（Uniform Resource Identifier）标准，将三层架构的所有可寻址资源（Identity、Room、Message、Channel、Thread、Socialware 资源等）映射为标准化的外部标识符。URI scheme 作为 Bus 层基础设施，是 Deep Link、资源分享、跨实例引用等上层能力的寻址基础。

## Motivation

### 现状

ezagent 当前有两套互不相通的寻址体系：

1. **Zenoh key_pattern**（Bus 内部）：`ezagent/{room_id}/index/{shard_id}/state`——面向 CRDT 同步引擎，暴露了 shard、state/updates 等实现细节，不适合用户层使用。
2. **HTTP API 路径**（产品层）：`/api/rooms/{room_id}/messages/{ref_id}`——面向本地 REST 客户端，无跨实例寻址能力（缺少 authority 部分）。

缺少的是一个**面向用户和应用的全局寻址标准**：给定一个 URI，任何 ezagent 客户端都能定位到对应的资源，无论资源在哪个实例、哪个 Relay 域。

### 如果不做

- 每个需要"打开 ezagent 中某个东西"的功能（分享、邀请、Deep Link、@mention 跳转）都要自己发明 URL 格式
- 跨实例引用没有标准表示（如何在 Room A 中引用 Room B 的一条消息？）
- 移动端/桌面端的 scheme 注册（`ezagent://`）没有规范，各客户端实现不一致

### 设计目标

1. **与三层架构对齐**：Mid-layer 四原语各有一级路径段
2. **P2P-First**：URI 不假设 Relay 在线，只将 relay_domain 作为命名空间
3. **渐进可解析**：有 ezagent 客户端时走 native scheme，无客户端时通过 Web fallback 可读
4. **可扩展**：Extension 和 Socialware 可在约定的子路径下注册自己的 URI 空间

## Design

### D1. URI 结构

```
ezagent://{authority}/{path}[?{query}][#{fragment}]
```

**各部分定义：**

| 部分 | 格式 | 来源 | 说明 |
|------|------|------|------|
| **Scheme** | `ezagent` | 固定 | IANA 注册（未来），客户端 scheme handler 注册 |
| **Authority** | `{relay_domain}` | Entity ID 的 relay_domain 部分 | 命名空间标识，不是网络地址 |
| **Path** | 见 §D2 | 资源类型 + 标识符 | 分层路径 |
| **Query** | `key=value&...` | 可选 | 过滤、分页、时间范围等 |
| **Fragment** | `#{anchor}` | 可选 | 客户端行为提示（如高亮某段内容） |

**Authority 语义**：

- Authority 就是 Entity ID 中 `@local:` 后面的 `relay_domain`
- [MUST] Authority 是**身份命名空间**，不是网络地址。客户端解析 URI 时，通过 DNS 解析 relay_domain 获取 Relay 地址，再通过 Relay 发现资源所在的 peer
- [MUST] 同一资源在不同网络位置（LAN 直连 / 经 Relay 中转）时，URI 不变

### D2. Path 映射

#### Mid-layer 四原语

| 资源 | Path | 标识符来源 |
|------|------|-----------|
| Identity | `/@{entity_id}` | architecture §1.4 |
| Room | `/r/{room_id}` | bus-spec §5.2, UUIDv7 |
| Message | `/r/{room_id}/m/{ref_id}` | bus-spec §5.3, ULID |
| Timeline | `/r/{room_id}/t` | bus-spec §5.3 |

**Identity path 说明**：entity_id 已包含 relay_domain（`@alice:relay.example.com`），URI 中的 authority 与 entity_id 中的 relay_domain 一致时可省略重复：

```
完整形式：  ezagent://relay.example.com/@alice:relay.example.com
规范形式：  ezagent://relay.example.com/@alice
跨域引用：  ezagent://relay-a.com/@bob:relay-b.com    （Bob 注册在 relay-b，但通过 relay-a 发现）
```

- [MUST] 当 entity_id 的 relay_domain 与 URI authority 一致时，Path 中 SHOULD 省略 `:relay_domain` 后缀
- [MUST] 当 entity_id 的 relay_domain 与 URI authority 不一致时（跨域），Path 中 MUST 保留完整 entity_id

#### Extension 资源

Extension 在 Mid-layer 路径下扩展子路径。命名空间为 Extension ID：

| 资源 | Path | Extension |
|------|------|-----------|
| Channel 视图 | `/r/{room_id}/c/{channel_name}` | EXT-04 Channels |
| Thread | `/r/{room_id}/m/{ref_id}/thread` | EXT-11 Threads |
| Reactions | `/r/{room_id}/m/{ref_id}/reactions` | EXT-03 Reactions |
| Profile | `/@{entity_id}/profile` | EXT-13 Profile |
| Media | `/r/{room_id}/blob/{blob_id}` | EXT-10 Media |

- [MUST] Extension URI path MUST 在其所增强的 Mid-layer 资源路径之下
- [MUST] Extension 注册 URI 子路径时，MUST 在 manifest.toml 中声明 `uri_paths`

#### Socialware 资源

Socialware 资源本质上是特定 content_type 的 Message，通过 namespace 子路径访问：

```
/r/{room_id}/sw/{namespace}/{resource_type}/{ref_id}
```

| 资源 | Path | Socialware |
|------|------|------------|
| TaskArena 任务 | `/r/{room_id}/sw/ta/task/{ref_id}` | TaskArena |
| EventWeaver 事件 | `/r/{room_id}/sw/ew/event/{ref_id}` | EventWeaver |
| ResPool 配额 | `/r/{room_id}/sw/rp/quota/{ref_id}` | ResPool |
| Agent 实例 | `/r/{room_id}/sw/af/agent/{agent_name}` | AgentForge |
| SW 整体视图 | `/r/{room_id}/sw/{namespace}` | 任意 Socialware |

- [MUST] Socialware URI path MUST 以 `/sw/{namespace}` 为前缀
- [MUST] `{namespace}` MUST 与 manifest.toml 中的 `namespace` 字段一致
- [SHOULD] Socialware 在 manifest.toml 中声明 `uri_resources`，列出可寻址的资源类型

#### 保留路径

| Path 前缀 | 用途 | 注册者 |
|-----------|------|--------|
| `/s/{token}` | Share Link | EEP-0003 (EXT-19 Share) |
| `/invite/{token}` | Room 邀请 | 未来 EEP |
| `/federation/` | 联邦协商 | 未来 EEP |

- [MUST] 以上保留前缀 MUST NOT 被 Extension 或 Socialware 注册使用

### D3. Query 参数约定

| 参数 | 适用资源 | 说明 | 示例 |
|------|---------|------|------|
| `after` | Timeline | ISO 8601 时间下界 | `?after=2026-03-01T00:00:00Z` |
| `before` | Timeline | ISO 8601 时间上界 | `?before=2026-03-02T00:00:00Z` |
| `limit` | Timeline | 最大返回条数 | `?limit=50` |
| `channel` | Timeline | 过滤特定 Channel | `?channel=code-review` |
| `content_type` | Timeline | 过滤特定消息类型 | `?content_type=ta:task.*` |

- [MAY] Extension 和 Socialware MAY 定义额外的 query 参数
- [MUST] 自定义 query 参数 MUST 以 namespace 为前缀：`?ta.status=open`

### D4. Web Fallback

当接收方没有安装 ezagent 客户端时，通过 HTTPS 访问 Relay 的 Web 端点获取可读视图：

```
ezagent URI:     ezagent://relay.example.com/r/{room_id}/m/{ref_id}
Web fallback:    https://relay.example.com/e/r/{room_id}/m/{ref_id}
```

**映射规则**：

```
ezagent://{authority}/{path}  →  https://{authority}/e/{path}
```

- [MUST] Web fallback path MUST 以 `/e/` 为前缀，与 Relay 的 Admin API (`/admin/`) 和 Health (`/health`) 路径区分
- [SHOULD] Relay 收到 Web fallback 请求时，SHOULD 返回 HTML 页面，包含：
  - 资源的只读预览（如消息内容、Room 信息）
  - "在 ezagent 中打开" 的 `ezagent://` deep link
  - 如果资源需要认证，提示用户登录或安装客户端
- [MAY] Relay MAY 对未认证的 Web fallback 请求返回受限视图（如仅显示 Room 名称和成员数，不显示消息内容）

### D5. URI 解析流程

```
                    ezagent://relay.example.com/r/{room_id}/m/{ref_id}
                                    │
                                    ▼
                    ┌─ 客户端解析 ─────────────────────┐
                    │ 1. 提取 authority: relay.example.com │
                    │ 2. 解析 path: /r/{room_id}/m/{ref_id}│
                    │ 3. 判断资源类型: Message             │
                    └────────────────────────────────────┘
                                    │
                        ┌───────────┴───────────┐
                        ▼                       ▼
                 本地缓存命中？              联系 Relay
                   是 → 直接展示          DNS 解析 relay_domain
                                              │
                                              ▼
                                    Relay 返回资源位置
                                    (room 所在 peer 列表)
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                              P2P 直连              Relay 中转
                           (同网段/已知 peer)     (跨网/peer 离线)
                                    │                   │
                                    └─────────┬─────────┘
                                              ▼
                                       获取资源数据
                                      展示给用户
```

- [MUST] 客户端 MUST 先检查本地是否已有该资源的 CRDT 数据
- [SHOULD] 客户端 SHOULD 优先尝试 P2P 直连获取数据
- [MAY] 如果 P2P 不可达，客户端 MAY 通过 Relay 获取 Cold Storage 中的数据副本

### D6. URI 注册机制

Extension 和 Socialware 通过 manifest.toml 声明自己的 URI 子路径：

**Extension manifest.toml**：

```toml
[extension]
id = "channels"

[uri]
paths = [
  { pattern = "/r/{room_id}/c/{channel_name}", description = "Channel view" }
]
```

**Socialware manifest.toml**：

```toml
[socialware]
id = "task-arena"
namespace = "ta"

[uri]
resources = [
  { type = "task",       pattern = "/sw/ta/task/{ref_id}",   description = "Single task" },
  { type = "submission", pattern = "/sw/ta/submit/{ref_id}", description = "Task submission" }
]
# 完整路径 = /r/{room_id} + resource.pattern
```

- [MUST] Engine 在加载 Extension/Socialware 时 MUST 检查 URI path 冲突
- [MUST] 两个 Extension 声明相同 path pattern 时，Engine MUST 拒绝加载后注册者并报错

### D7. 规范化规则

| 规则 | 示例 |
|------|------|
| Authority 小写 | `Relay.Example.COM` → `relay.example.com` |
| Path 保留大小写 | `/r/019a3B4c-...` 保持原样（UUIDv7 是 hex） |
| 尾部斜杠去除 | `/r/{room_id}/` → `/r/{room_id}` |
| 空 query 省略 | `/r/{room_id}?` → `/r/{room_id}` |
| Entity ID 中 `@` 保留 | `/@alice` 中 `@` 不做 URL 编码 |

- [MUST] URI 比较时 MUST 先规范化再比较
- [MUST] `room_id`（UUIDv7）和 `ref_id`（ULID）MUST 使用原始大小写（不做 case-folding）

## Alternatives Considered

### A1. 使用 HTTP URL 作为唯一标识

不定义 `ezagent://` scheme，统一使用 `https://{relay}/e/...`。

**优点**：无需 scheme 注册，任何浏览器可直接打开。

**放弃原因**：强依赖 Relay 在线（违反 P2P-First）；无法区分"在浏览器中查看"和"在 ezagent 客户端中打开"；与 Entity ID 中 relay_domain 是"命名空间而非地址"的语义不一致。

### A2. 使用 `matrix.to` 式中间服务

引入 `ezagent.to` 中心化跳转服务，将 `https://ezagent.to/#/@alice:relay.example.com` 重定向到客户端。

**优点**：统一入口，不依赖各 Relay 的 Web 实现。

**放弃原因**：引入中心化依赖（Matrix 社区已在讨论去除 matrix.to 依赖）；增加运营成本；可作为未来可选优化，但不应是协议层设计。

### A3. 在 Zenoh key_pattern 上直接暴露

将 `ezagent/{room_id}/content/{id}/state` 格式直接作为 URI。

**放弃原因**：暴露 CRDT 实现细节（shard_id、state/updates）；用户不可读；key_pattern 变更会导致所有 URI 失效。

## Implementation Strategy

| 阶段 | 内容 | 依赖 |
|------|------|------|
| **Stage 1** ✅ | architecture.md 新增 §1.5 URI Scheme 定义 | 本 EEP Accepted |
| **Stage 2** ✅ | extensions-spec / socialware-spec manifest 格式新增 `[uri]` | Stage 1 |
| **Stage 3** | 客户端 `ezagent://` scheme 注册 + URI 解析器 | Phase 3 (CLI) / Phase 4 (App) |
| **Stage 4** | Relay Web Fallback 端点 | relay-spec 更新 |

**验收标准**：

- [ ] 三层架构每个可寻址资源都有唯一 URI 表示
- [ ] `ezagent open {uri}` CLI 命令可跳转到对应资源
- [ ] Web Fallback URL 可在浏览器中打开并展示只读预览
- [ ] 两个不同的 Extension 声明相同 URI path 时，Engine 报错

## Open Questions

### Q1. IANA Scheme 注册

`ezagent://` 是否需要正式向 IANA 注册？目前可作为 provisional scheme 使用，正式注册可在协议稳定后进行。

### Q2. URI 中的 Room 可读性

当前 Room 使用 UUIDv7（`/r/019a3b4c-...`），不可读。是否支持 Room alias 作为替代路径？

```
规范形式：  ezagent://relay.example.com/r/019a3b4c-...
别名形式：  ezagent://relay.example.com/r/feature-review    （alias → room_id 映射存储在 Room Config）
```

如支持，需要定义 alias 唯一性约束和解析优先级。

### Q3. 版本化

URI scheme 是否需要版本号（如 `ezagent-v1://`）？

倾向不需要——通过 Path 的可扩展性处理未来变更，而非 scheme 版本化。Scheme 变更意味着所有旧链接失效，代价过高。

### Q4. URI 长度限制

Socialware 的深层嵌套路径（`/r/{room_id}/sw/ta/task/{ref_id}`）加上 query 参数可能导致 URI 很长。是否需要定义长度上限或短链接机制？
