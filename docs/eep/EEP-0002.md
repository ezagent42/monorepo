# EEP-0002 — Bridge Extension (EXT-18)

> **EEP**: 0002
> **标题**: Bridge Extension for External Platform Interop
> **类型**: Standards
> **状态**: Draft
> **作者**: Allen
> **创建日期**: 2026-02-28
> **关联 Spec 版本**: v0.9.5
> **目标 Spec 版本**: TBD（建议 v0.11+ / Phase 4 之后）
> **依赖 EEP**: [EEP-0001](EEP-0001.md)（ezagent URI Scheme）

---

## Abstract

提出 EXT-18 Bridge Extension，为 ezagent 提供与外部通信平台（Slack、Email、Discord、Mattermost 等）的双向消息桥接能力。Bridge 在 Extension 层定义 Bridged Entity 身份模型、消息路由 Hook 和归一化消息格式，同时通过 Connector Plugin 接口将平台特定协议隔离到独立可替换的 Python 模块中。

## Motivation

### 现状问题

ezagent 协议目前没有与外部通信平台互通的标准化机制。当 Socialware（如 TaskArena）需要将任务通知发送到 Slack channel，或接收来自 Email 的提交时，面临以下困境：

1. **身份映射无协议层支持**。外部用户没有 ezagent Entity ID。Architecture §1.4 的 Entity ID 体系 (`@local:relay_domain`) 完全面向原生节点，没有为外部 identity 提供映射层。
2. **Socialware 重复造轮子**。TaskArena 和 ResPool 各自对接 Slack 时，都要独立实现 OAuth、Webhook 解析、消息格式转换——违反 "Extension 做能力，Socialware 做业务" 的分层哲学。
3. **消息语义丢失**。外部消息存为普通 `immutable` 文本，丢失来源元信息。其他 Hook 无法基于来源做差异化处理。
4. **Hook Pipeline 无法统一处理外部流量**。桥接逻辑散落在各 Socialware 中，pre_send 阶段无法统一验证，after_write 阶段无法统一路由。

### 不做会怎样

短期可行但长期不可持续。随着 Socialware 数量增长和对接平台增多，产生 N×M 组合爆炸（N 个 Socialware × M 个平台），每个组合重复实现相同的桥接基础设施。

### 与 AgentForge Adapter 的关系

**Bridge 和 AgentForge Adapter 是不同层面的抽象，不应合并：**

| 维度 | Bridge Connector | AgentForge Adapter |
|------|-----------------|-------------------|
| 桥接对象 | 外部平台的 **用户 + 消息** | 外部 AI 的 **推理能力** |
| Identity 语义 | 外部用户 → Bridged Entity | LLM 无 Identity，Agent 才有 |
| 数据流 | 双向对等（消息级 1:1） | 请求-响应（N:1 聚合） |
| 外部服务角色 | 对等平台（有用户体系） | 无状态工具（提供计算） |

**边界案例——外部 Agent 平台**（如 LangGraph Cloud）的推荐处理：Bridge 负责传输层（创建 Bridged Entity），AgentForge 负责编排层（赋予 Role/Flow）。两者通过 Entity ID 和 EXT-13 Profile `entity_type: bridged_agent` 协作。

## Design

### D1. Bridged Entity 模型

外部用户在 ezagent 中以 Bridged Entity 身份存在：

```
Entity ID 格式: @{connector}_{remote_id}:{relay_domain}
示例:          @slack_U04ABCDEF:relay.example.com
               @email_bob_at_corp.com:relay.example.com
```

**身份注册**：Bridge Framework 通过标准 Identity 注册流程在 Relay 上创建 Bridged Entity。Bridged Entity 的密钥对由 Bridge Framework 持有（代理签名）。

**Profile 标记（EXT-13）**：

```yaml
entity_type: "bridged"
ext.bridge:
  connector: "slack"                    # Connector 类型
  remote_id: "U04ABCDEF"               # 外部平台用户 ID
  remote_display_name: "Bob Chen"       # 外部显示名
  remote_avatar_url: "https://..."      # 外部头像
  remote_profile_url: "https://team.slack.com/team/U04ABCDEF"  # 外部平台个人主页
  bridged_by: "@bridge-svc:relay.example.com"  # Bridge 服务 Entity
  bridged_at: "2026-03-15T08:00:00Z"   # 桥接时间
```

- [MUST] Bridged Entity 的 `entity_type` 字段 MUST 为 `"bridged"`。
- [MUST] Bridged Entity 的所有 Message MUST 由 Bridge Framework 代理签名。
- [MUST] 客户端 SHOULD 在 UI 上对 Bridged Entity 显示平台标识（如 Slack 图标）。

**URI 寻址（EEP-0001）**：

Bridged Entity 自动获得标准 ezagent URI：

```
ezagent://relay.example.com/@slack_U04ABCDEF

Web Fallback: https://relay.example.com/e/@slack_U04ABCDEF
→ 预览页显示外部平台标识 + 原始 Profile 信息
```

- [MUST] Bridged Entity 的 URI 遵循 EEP-0001 Identity path 规则，无需特殊处理
- [SHOULD] Web Fallback 页面 SHOULD 显示外部平台来源标识和 `remote_profile_url`（如 Slack 个人主页链接）

### D2. Room Config 字段

```yaml
ext.bridge:
  routes:
    - connector: "slack"
      remote_channel: "C04XYZ789"       # Slack channel ID
      remote_uri: "https://team.slack.com/archives/C04XYZ789"  # 外部平台可访问 URL
      direction: "bidirectional"         # bidirectional | inbound_only | outbound_only
      filters:                           # 可选过滤
        content_types: ["immutable", "mutable"]  # 仅桥接这些类型
        channels: ["general"]            # 仅桥接这些 ezagent channel
    - connector: "email"
      remote_channel: "dev-team@corp.com"
      remote_uri: "mailto:dev-team@corp.com"
      direction: "outbound_only"
```

- [MUST] `ext.bridge` 的 writer_rule: `signer.power_level >= admin`。
- [MUST] 未安装 Bridge Extension 的 Peer 收到含 `ext.bridge` 的 Room Config 时，按 §3.3.1 保留规则保留字段。

### D3. Ref Annotation

桥接消息在 Ref 上携带来源元信息和**双向 URI 映射**：

```yaml
# Inbound（外部→ezagent）的 Ref
ext.bridge:
  source: "slack"
  remote_message_id: "1700000000.000100"
  remote_channel: "C04XYZ789"
  remote_timestamp: "2026-03-15T08:30:00Z"
  remote_uri: "https://team.slack.com/archives/C04XYZ789/p1700000000000100"

# Outbound（ezagent→外部）成功后的 Annotation 回写
ext.bridge:
  deliveries:
    - connector: "slack"
      remote_message_id: "1700000001.000200"
      remote_uri: "https://team.slack.com/archives/C04XYZ789/p1700000001000200"
      delivered_at: "2026-03-15T08:30:05Z"
      status: "delivered"
```

**双向 URI 映射**：

`remote_uri` 字段建立了 ezagent URI ↔ 外部平台 URL 的双向关系：

| 方向 | 场景 | 流程 |
|------|------|------|
| 外部 URL → ezagent URI | 用户在 ezagent 中粘贴 Slack 消息链接 | Connector.`resolve_remote_uri()` 解析 URL → 查找对应 ref_id → 跳转到本地消息 |
| ezagent URI → 外部 URL | 在外部平台引用 ezagent 消息 | 从 Ref Annotation 读取 `remote_uri` → 直接使用；或 Connector.`build_remote_uri()` 动态构建 |

- [MUST] Inbound Ref 的 `ext.bridge.source` 字段 MUST 标明来源 Connector。
- [SHOULD] Inbound Ref SHOULD 携带 `remote_uri`，由 Connector 在注入时提供。
- [MUST] Outbound delivery 状态 MUST 通过 Annotation 回写，不修改原始 Ref。
- [SHOULD] Outbound delivery Annotation SHOULD 携带 `remote_uri`，指向外部平台中的投递结果。
- [MUST] Bridge Framework MUST 维护 `remote_message_id ↔ ref_id` 幂等映射，防止重复注入。
- [MUST] Bridge Framework SHOULD 维护 `remote_uri ↔ ezagent URI` 索引，支持双向查找。

### D4. Hook 定义

| Hook | Phase | Trigger | Priority | 说明 |
|------|-------|---------|----------|------|
| `bridge.validate_inbound` | pre_send | `ext.bridge.source != null` | 30 | 验证 inbound 消息合法性（签名、Bridged Entity 有效性） |
| `bridge.resolve_uri` | after_read | 消息体中包含外部平台 URL | 35 | 检测消息中的外部 URL，匹配已安装 Connector 的 `resolve_remote_uri()`，为匹配的 URL 附加 ezagent URI 跳转 |
| `bridge.route_outbound` | after_write | `timeline_index.insert` | 80 | 匹配 Room Config 的 `ext.bridge.routes`，异步转发到目标 Connector |
| `bridge.delivery_report` | after_write | delivery 完成 | 85 | 将 delivery 状态回写为 Annotation |

**Priority 设计理由**：

- Inbound 验证（30）在 Identity 签名验证（0）之后、Socialware Hook（≥100）之前
- Outbound 路由（80）在标准 Extension Hook 范围内、Socialware Hook 之前
- 这确保 Socialware 看到的消息已经过 Bridge 验证和标注

**after_write 异步约束**：

- [MUST] `bridge.route_outbound` 中的外部 API 调用 MUST 通过 `tokio::spawn_blocking` 异步执行，遵循 bus-spec §3.2 的规定。
- [MUST] 外部 API 调用失败 MUST NOT 影响已 apply 的 CRDT 数据。失败通过 delivery_report Annotation 记录。

### D5. Connector Plugin 接口

Bridge Framework 通过 PyO3 加载 Python Connector Plugin，与 Socialware Runtime 使用相同的调用机制。

```python
class BridgeConnector(Protocol):
    """每个 Connector = 一个外部平台的全部桥接逻辑"""

    # ── 元信息 ──
    def get_meta(self) -> ConnectorMeta: ...
    def get_capabilities(self) -> set[Capability]: ...
        # Capability: TEXT, RICH_TEXT, MEDIA, REACTIONS,
        #   THREADS, PRESENCE, TYPING, READ_RECEIPTS, EDIT, DELETE

    # ── 认证 ──
    def get_auth_flows(self) -> list[AuthFlow]: ...
    async def authenticate(self, flow_id: str,
                          credentials: dict) -> AuthSession: ...

    # ── 生命周期 ──
    async def start(self, config: ConnectorConfig,
                    sink: InboundSink) -> None: ...
    async def stop(self) -> None: ...

    # ── Outbound ──
    async def send_message(self, target: RemoteTarget,
                          message: NormalizedMessage) -> RemoteMessageID: ...
    async def edit_message(self, remote_id: RemoteMessageID,
                          content: NormalizedMessage) -> None: ...
    async def delete_message(self, remote_id: RemoteMessageID) -> None: ...
    async def send_reaction(self, remote_id: RemoteMessageID,
                           emoji: str) -> None: ...

    # ── Identity 解析 ──
    async def resolve_user(self, remote_user_id: str) -> BridgedUserInfo: ...
    async def resolve_channel(self, remote_channel_id: str) -> BridgedChannelInfo: ...

    # ── URI 映射（EEP-0001）──
    def resolve_remote_uri(self, remote_uri: str) -> RemoteResource | None:
        """解析外部平台 URL，返回对应的 remote_id 和资源类型。
        用于：用户在 ezagent 中粘贴外部链接时，自动识别为已桥接的资源。
        返回 None 表示该 URL 不属于本 Connector 管辖。"""
        ...

    def build_remote_uri(self, resource_type: str,
                        remote_id: str) -> str | None:
        """将 remote_id 构建为外部平台可访问的 URL。
        用于：Inbound/Outbound 消息的 ext.bridge.remote_uri 字段填充。
        返回 None 表示该平台不支持 URL 构建（如某些 API-only 平台）。"""
        ...


@dataclass
class RemoteResource:
    """resolve_remote_uri 的返回类型"""
    connector: str                      # Connector ID
    resource_type: str                  # "message" | "channel" | "user" | "thread"
    remote_id: str                      # 外部平台资源 ID
    remote_channel: str | None = None   # 所属 channel（如适用）


@dataclass
class NormalizedMessage:
    """平台无关的归一化消息格式"""
    text: str                           # 纯文本 fallback
    rich_text: str | None = None        # Markdown
    media: list[MediaRef] = field(default_factory=list)
    reply_to: RemoteMessageID | None = None
    thread_root: RemoteMessageID | None = None
    metadata: dict = field(default_factory=dict)  # 平台特定透传


class InboundSink(Protocol):
    """Connector 通过此接口向 Bridge Framework 注入外部事件"""
    async def on_message(self, source: RemoteSource,
                        message: NormalizedMessage) -> None: ...
    async def on_edit(self, remote_id: RemoteMessageID,
                     content: NormalizedMessage) -> None: ...
    async def on_delete(self, remote_id: RemoteMessageID) -> None: ...
    async def on_reaction(self, remote_id: RemoteMessageID,
                         user: str, emoji: str) -> None: ...
    async def on_typing(self, source: RemoteSource) -> None: ...
    async def on_presence(self, user: str, status: str) -> None: ...
```

### D8. CLI 命令

```bash
# Connector 管理
ezagent bridge list-connectors           # 列出已安装 Connector
ezagent bridge install slack             # 安装 Connector
ezagent bridge uninstall slack           # 卸载 Connector

# Room 路由管理
ezagent bridge add-route ROOM_ID \
  --connector slack \
  --remote-channel C04XYZ789 \
  --direction bidirectional

ezagent bridge remove-route ROOM_ID --connector slack
ezagent bridge list-routes ROOM_ID

# Bridged Entity 管理
ezagent bridge list-entities              # 列出所有 Bridged Entity
ezagent bridge entity-info @slack_U04ABC  # 查看详情

# 认证
ezagent bridge auth slack                 # 交互式认证流程

# URI 跳转（EEP-0001）
ezagent open ezagent://relay.example.com/@slack_U04ABCDEF
  # → 打开 Bridged Entity Profile（含外部平台链接）

# 外部 URL 解析
ezagent bridge resolve-url "https://team.slack.com/archives/C04XYZ789/p1700000000"
  # → 输出: ezagent://relay.example.com/r/{room_id}/m/{ref_id}
  #   (如果该消息已桥接)
```

## Alternatives Considered

### A1. 纯 Socialware 实现（不加 Extension）

在 Socialware 层实现桥接逻辑，每个需要外部交互的 Socialware 自行处理。

**优点**：零架构冲击。

**放弃原因**：N×M 组合爆炸；身份映射无协议层保证；Hook 优先级在 Socialware 范围（≥100），无法在 Extension 层统一验证外部消息；违反分层哲学。

### A2. Relay 层实现

在 Relay 服务中实现桥接，类似 Matrix 的 Application Service 模型。

**优点**：中心化管理，Bridged Entity 身份由 Relay 直接控制。

**放弃原因**：违反 P2P-First 原则——Bridge 只能在有 Relay 的场景下工作；增加 Relay 的职责和复杂度（Relay 设计原则是"邮递员，不是房东"）；需要 Relay 理解外部平台协议，超出其当前纯 Rust 无 Python 的技术栈。

### A3. 合并 Bridge 与 AgentForge Adapter 为统一接口

设计一个包含 Identity 映射和 AI 调用的统一外部服务接口。

**优点**：概念统一。

**放弃原因**：违反接口隔离原则（ISP）——Adapter 不需要 Identity 映射和双向路由，Bridge 不需要 Conversation Segment 和 Lifecycle 状态机。合并后每个实现 stub 一半接口。两者桥接的"东西"不同（Identity+Message vs Intelligence），强行统一会模糊语义。

## Implementation Strategy

### 优先级与阶段

Bridge Extension **不应进入 Phase 1–2**。建议的实施时间线：

| 阶段 | 内容 | 前置依赖 |
|------|------|---------|
| **Pre-work** | EEP 评审通过，Spec 定稿 | 本 EEP Accepted + EEP-0001 Accepted |
| **Stage 1: Framework** | Bridge Extension Rust 核心（Hook 注册、Route 匹配、Entity 管理、URI path 注册）| Phase 2 完成（EXT-13 Profile 可用）|
| **Stage 2: Connector Interface** | Python BridgeConnector Protocol + InboundSink | Phase 4 完成（CLI + HTTP 可用）|
| **Stage 3: Reference Connector** | 一个完整的 Connector 实现（建议 Slack 或 Email） | Stage 2 |
| **Stage 4: CLI + HTTP** | `bridge` 命令组 + REST 端点 | Stage 2 + Phase 4 |
| **Stage 5: UI** | Chat App 的 Bridge 管理界面和 Bridged Entity 渲染 | Stage 3 + Phase 5 |

### 验收标准

- [ ] Bridged Entity 可在 Room 中正常显示，UI 标明外部来源
- [ ] Bridged Entity 的 ezagent URI 可正常解析和跳转（EEP-0001）
- [ ] Inbound 消息通过完整 Hook Pipeline，签名验证通过
- [ ] Inbound Ref 的 `ext.bridge.remote_uri` 正确填充
- [ ] Outbound 消息异步投递，delivery 状态（含 `remote_uri`）回写为 Annotation
- [ ] 外部平台 URL 可通过 `bridge.resolve_uri` Hook 或 CLI/HTTP 解析为 ezagent URI
- [ ] Connector 安装/卸载不影响 Bridge Framework 核心
- [ ] 至少一个 Reference Connector（Slack 或 Email）通过集成测试
- [ ] 幂等性：重复的外部事件不产生重复 Ref

## Open Questions

### Q1. Bridged Entity 的密钥管理

Bridge Framework 为 Bridged Entity 代理签名。密钥存储在哪里？

- **选项 A**：Bridge Framework 的本地 keystore（`~/.ezagent/bridge/keys/`）
- **选项 B**：Relay 托管密钥（Bridge 请求 Relay 代签）
- **选项 C**：每个 Connector 独立管理密钥

倾向 A，但需要讨论密钥轮换和多节点同步场景。

### Q2. Bridged Entity 的 Quota 策略

Bridged Entity 在 Relay 上消耗 Quota 应归属谁？

- **选项 A**：归属创建 Bridge 的管理员 Entity
- **选项 B**：Bridged Entity 独立计 Quota
- **选项 C**：Bridge 服务 Entity 统一计 Quota

### Q3. 消息格式转换的精度

`NormalizedMessage` 是否足够覆盖主流平台的消息特性？

需要对 Slack Block Kit、Discord Embed、Email MIME、企业微信消息格式做逐一映射验证。可能需要 `metadata: dict` 透传字段来保留无法归一化的平台特定数据。

### Q4. Compliance Level 归属

EXT-18 应归入现有 Level 3: Full，还是新增 Level 4: Interop？

新增 Level 的好处是不增加 Full compliance 的负担；坏处是层级过多增加理解成本。

### Q5. 与 Federation 的关系

当两个 ezagent 集群需要互通时（跨 Relay），是走 Bridge 还是走 Relay 联邦机制？

初步判断：同协议互通走 Relay 联邦（bus-spec §6.5），异协议互通走 Bridge。但边界需要明确。

### Q6. 外部 URL 解析的触发时机

`bridge.resolve_uri` Hook（after_read, priority 35）会扫描消息体中的外部 URL 并尝试匹配已桥接资源。这涉及几个问题：

- **性能**：每条消息都扫描 URL 是否太重？是否需要限制为仅在用户显式请求时才解析（如右键菜单 "Resolve external link"）？
- **多 Connector 冲突**：一个 URL 可能被多个 Connector 的 `resolve_remote_uri()` 匹配。是否需要 priority 机制？还是 first-match-wins？
- **缓存**：`remote_uri ↔ ezagent URI` 映射索引的存储位置和失效策略？

### Q7. 与 EEP-0003 Share 的交叉

当 Share 的 target 包含桥接消息（来自 Bridged Entity）时：
- 接收方未安装 Bridge Connector——`remote_uri` 可作为 fallback 链接直接打开外部平台
- 接收方已安装 Bridge Connector——可直接在 ezagent 中查看，Bridged Entity 正常显示
- Share 的 `@shareable.transform` 是否需要对 `ext.bridge` Annotation 做特殊处理（如脱敏 remote_uri）？
